# SENG 468 – Distributed Semantic Retrieval System

A scalable semantic search engine against user-uploaded PDF documents.

Allows multiple concurrent uploads/searches using async processing + Dockerized deployment

## Features

- Authentication (JWT, HS256, expiry 24h)
- Asynchronous PDF upload + background processing (responds with 202 right away)
- PDF parsing/embedding generation worker via RabbitMQ
- Vector similarity search (top 5 paragraphs)
- User isolation, users can only see documents they uploaded
- Redis cache for query embeddings (LRU eviction policy, 256 MB)
- Horizontally scaled (Nginx load balancer distributing requests between 2 API instances)
- PostgreSQL connection pooling (`psycopg-pool`)
- Zero-config deployment (`docker compose up`)

## Architecture Overview

```
Client → nginx (load balancer, port 8080, least-conn)
             ├→ api1 (Flask + Gunicorn, 4 workers)
             └→ api2 (Flask + Gunicorn, 4 workers)
                  ├→ PostgreSQL (users, auth)
                  ├→ MinIO (stores raw PDFs as objects)
                  ├→ Redis (query embedding cache)
                  ├→ Qdrant (vector search)
                  └→ RabbitMQ ←→ Worker1/Worker2 → Qdrant
```

**nginx** – Serves as a load balancer between api1/api2. Distributes client requests using least-connection policy.

**API (api1, api2)** – Stateless Flask + Gunicorn REST API servers. Handles auth, pdf uploads, search queries. Embedding model loaded into memory once on server startup (via `--preload` arg) and shared among workers.

**Worker (worker1, worker2)** – Dedicated workers listening to jobs published on RabbitMQ. Parse/chunk uploaded PDFs into paragraphs → generate embeddings → upsert into Qdrant search engine. Two instances share the same queue for parallel processing.

**PostgreSQL** – Database service storing user account information. Connection pooled in API service (2-20 connections).

**MinIO** – S3-compatible object storage service. Stores uploaded PDF documents. Keeps API service stateless (does not store anything locally on disk).

**Qdrant** – Vector database used for similarity search. Stores embedding vectors for every paragraph generated by worker. Utilizes HNSW index. Cosine similarity search automatically filters search results by `user_id` payload.

**RabbitMQ** – Persistent message broker. Decouples user upload requests from processing workload. Worker processes can be independently scaled up/down.

**Redis** – Query cache. Stores computed embedding vectors keyed by SHA256 hash of the query string. Able to skip expensive ~80ms embedding generation step on repeat searches.

All services bootstrapped with Docker Compose.

## API Endpoints (nginx port 8080)

### Authentication

#### POST /auth/signup

Request:

```json
{ "username": "alice", "password": "securepassword123" }
```

Response:

- `200` → `{ "message": "User created successfully", "user_id": "uuid" }`
- `409` → `{ "error": "Username already exists" }`

#### POST /auth/login

Request:

```json
{ "username": "alice", "password": "securepassword123" }
```

Response:

- `200` → `{ "token": "eyJhbG...", "user_id": "uuid" }`
- `401` → `{ "error": "Invalid credentials" }`

All authed endpoints require this token passed as such:

```
Authorization: Bearer <token>
```

### Documents

#### POST /documents (multipart/form-data, field: `file`)

Responds with **202 Accepted** immediately, uploads PDF in background job via `RabbitMQ`.

Response:

```json
{
  "message": "PDF uploaded, processing started",
  "document_id": "uuid",
  "status": "processing"
}
```

#### GET /documents

Lists current user's uploaded documents.

Response:

```json
[
  {
    "document_id": "uuid-123",
    "filename": "paper.pdf",
    "upload_date": "2026-03-15T10:30:00+00:00",
    "status": "ready",
    "page_count": 42
  }
]
```

- `status`: `"processing"` until worker has finished processing chunks, then set to `"ready"`
- `page_count`: number of paragraphs embedded into `Qdrant` (NOT actual PDF pages)

#### DELETE /documents/{id}

- Deletes PDF object from `MinIO`
- Deletes ALL embeddings from `Qdrant` database
- `200` → `{ "message": "Document and all associated data deleted", "document_id": "uuid-123" }`
- `404` → `{ "error": "Document not found or not owned by user" }`

### Search

#### GET /search?q=\<query\>

Response:

```json
[
  {
    "text": "Machine learning practitioners can ...",
    "score": 0.942,
    "document_id": "uuid-123",
    "filename": "paper.pdf"
  }
]
```

- Exactly 5 results (less if not enough data)
- Sorted by highest score first
- Scores ∈ [0, 1] (cosine similarity metric)
- Can only search against documents you uploaded
- Caches query embeddings on `Redis` (cache hit will skip ~80ms embedding generation step)

### Health

#### GET /health

Response:

```json
{ "status": "ok" }
```

Response **header** `X-Served-By: api1` or `api2` indicates which instance served the request. Useful for checking if load balancer is working.

## Processing Pipeline (Background)

1. Upload → Store PDF in `MinIO` bucket (key `user_id/doc_id/filename`)
2. Push job onto `RabbitMQ` queue (`pdf_processing`, persistent)
3. Return 202 Accepted to user/client immediately
4. Worker retrieves job from queue:
   1. Download PDF from `MinIO` bucket
   2. Parse PDF → text (uses `PyMuPDF`)
   3. Chunk into paragraphs (min 50 chars / max 1200 chars, 1-sentence overlap)
   4. Generate embedding (`all-MiniLM-L6-v2` base model, 384-dim, normalized)
   5. Upsert into `Qdrant` with payload: `user_id`, `document_id`, `filename`, `text`
5. `GET /documents` document status flips from `"processing"` → `"ready"` once worker has upserted paragraphs into `Qdrant`.

## Setup Guide

### 1. Clone repo, build & configure

```bash
git clone <your-git-url>
cd SENG-468-FINAL-PROJECT
cp .env.example .env
```

### 2. Build/start everything

```bash
docker compose up --build -d
```

First startup/build may take 5-10 minutes. Embedding model must be downloaded.
Thereafter starts will be fast.

### 3. Test it's working?

```bash
# wait until services have started
docker compose logs -f

# hit health endpoint
curl http://localhost:8080/health

# verify nginx load balancing (run concurrent requests, should see both APIs serving responses)
for i in $(seq 1 10); do
  curl -s -I http://localhost:8080/health | grep X-Served-By &
done
wait
```

### 4. Smoke Test (paste everything)

```bash
# signup
curl -X POST http://localhost:8080/auth/signup \
  -H "Content-Type: application/json" \
  -d '{"username":"alice","password":"securepassword123"}'

# login
TOKEN=$(curl -s -X POST http://localhost:8080/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"alice","password":"securepassword123"}' | python3 -c "import sys,json; print(json.load(sys.stdin)['token'])")

# upload pdf
curl -X POST http://localhost:8080/documents \
  -H "Authorization: Bearer $TOKEN" \
  -F "file=@tests/sample1.pdf"

# wait for upload & processing to complete/it usually dont take 100 but just to make sure 
sleep 100

# search!
curl "http://localhost:8080/search?q=test" \
  -H "Authorization: Bearer $TOKEN" | python3 -m json.tool
```

## Load Testing

CLI-based load testing suite (uses `curl` + bash background processes for concurrency, `python3` for stats).

```bash
bash load-test/run_loadtest.sh
```

Runs 5 different test scenarios with a 30-second warmup period before measurements begin.

| Scenario | What it tests           | Concurrent users |
|----------|-------------------------|------------------|
| 1        | Concurrent uploads      | 20 users × 3 PDFs|
| 2        | Concurrent searches     | 50 parallel      |
| 3        | Mixed workload          | 20 uploaders + 30 searchers|
| 4        | Steady state load       | 10 concurrent for 60s|
| 5        | Breaking point          | 25 → 50 → 100 → 200 |

To collect all results for your report:

```bash
bash load-test/collect_report_data.sh | tee report_data.txt
```

Run `run_loadtest.sh` first. This script gathers hardware specs, verifies load balancer health, and collects resulting metrics from all load test scenarios.

**Requirements from spec:**

| Endpoint          | Target |
|-------------------|--------|
| `POST /documents` | < 500 ms P95 |
| `GET /search`     | < 2 seconds P95 under concurrent load |
| Mixed workload    | Zero 5xx responses |

## Docker Compose Services

Spin everything up with...

```
docker compose up --build
```

| Service      | Image               | Role |
|--------------|---------------------|------|
| nginx        | nginx:1.25-alpine   | Load balancer (bound to port 8080) |
| api1         | Flask + Gunicorn    | API instance 1 |
| api2         | Flask + Gunicorn    | API instance 2 |

## Troubleshooting

**Searching returns no results?** The worker hasn't processed the PDF yet. View worker logs with `docker compose logs worker1` and wait until you see "Document xxx processed".

**Worker container logs show "Connection refused"?** That's expected on initial startup. `RabbitMQ` takes about ~15s to become available. Worker will retry up to 30x automatically.

**Encountering Docker permission errors? (UVic cluster)**

```bash
mkdir -p /tmp/<your-docker-data>
echo '{"data-root":"/tmp/<your-docker-data>"}' > ~/.config/docker/daemon.json
systemctl --user restart docker
```

**Clean restart:**

```bash
docker compose down -v
docker compose up --build -d
```

**View logs:**

```bash
docker compose logs -f          # follow logs for all services
docker compose logs -f worker1  # just worker1 service logs
docker stats                    # live resource usage for all containers
```
